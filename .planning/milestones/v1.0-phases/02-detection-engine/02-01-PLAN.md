## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LinkedIn DOM → content script | Untrusted text/HTML enters via postText extraction; must never be eval'd or innerHTML'd |
| LinkedIn DOM → href attribute | authorProfileUrl read from anchor.href; LinkedIn-controlled but treated as untrusted for downstream usage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering (XSS) | extractPostData postText read | mitigate | Use `.innerText` only; never `.innerHTML`. authorProfileUrl is read from `anchor.href` (browser-parsed URL), not from textContent of a span |
| T-02-02 | Information Disclosure | PostData containing postText in memory | accept | Memory-only per STATE.md; never written to chrome.storage.local (enforced by Plan 03 storage layer which only writes signalBreakdown, not postText) |
| T-02-03 | Tampering | Inline selector strings drift into observer.ts | mitigate | Observer imports POST_BODY_TEXT/POST_AUTHOR_LINK/RESHARE_INDICATOR from selectors.ts; CLAUDE.md constraint #1 (INFRA-04) preserved by registry-only grep gate in Task 1 verify |
| T-02-04 | Denial of Service (ReDoS) | authorId slug regex `\/in\/([^/?#]+)/` | mitigate | Bounded character class `[^/?#]+` with no nested quantifiers; non-backtracking on realistic URLs |
| T-02-SC | Tampering | Package installs in this plan | accept | No package-manager installs in Plan 01 (only TypeScript edits). Plan 02 handles fast-levenshtein with its own legitimacy gate per RESEARCH.md §Package Legitimacy Audit |
