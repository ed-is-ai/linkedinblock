## Integration Findings

### INT-01 — WARNING: Threshold slider does not update observer hide threshold in-session

| Attribute | Value |
|-----------|-------|
| Severity | Warning |
| File | `src/content/index.ts` line 296 |
| Issue | `effectiveHideThreshold` in observer closure reads `autoHideThreshold` captured at `init()` time, not `currentThreshold` module variable that is updated by the settings `onChanged` handler (line 169). Moving the threshold slider rebuilds `thresholdAuthors` correctly (fast-path for stored accounts), but the observer's score-based hide check at line 296 continues using the old threshold for newly-detected posts within the same session. |
| Impact | Newly-detected posts from accounts near the threshold boundary may not be hidden after a slider change until page reload. Stored accounts (in `thresholdAuthors`) are affected immediately. |
| Proposed fix | Replace `effectiveHideThreshold` constant in the observer closure with `currentThreshold` module variable at line 296 (requires removing the `const effectiveHideThreshold` declaration or reassigning from `currentThreshold`). |

### INT-02 — BLOCKER FOR PHASE 19: No `blockedAt` timestamp on FlaggedAccount

| Attribute | Value |
|-----------|-------|
| Severity | Blocker for Phase 19 BLOCK-01 |
| File | `src/shared/types.ts` (FlaggedAccount type), `src/popup/index.tsx` line 63 (handleBlock) |
| Issue | `handleBlock` writes `{ ...existing, status: 'blocked' as const }` with no timestamp. `FlaggedAccount` has no `blockedAt` field. BLOCK-01 requires each row in `blocked.html` to show "block date". |
| Phase 19 action | Add `blockedAt?: number` to `FlaggedAccount` in `types.ts`. Add `blockedAt: Date.now()` to the `handleBlock` write in `popup/index.tsx` line 63. |

### INT-03 — BLOCKER FOR PHASE 19: Content script has no `blocked→pending` reversal branch

| Attribute | Value |
|-----------|-------|
| Severity | Blocker for Phase 19 BLOCK-02 |
| File | `src/content/index.ts` lines 144–164 (flaggedAccounts onChanged handler) |
| Issue | The handler only handles new blocked-status upgrades (adds to `blockedAuthors`, shows red tombstone for newly-appearing posts). It has no branch for the reverse: when `status` transitions from `'blocked'` back to `'pending'` (an unblock). If Phase 19 simply writes `status: 'pending'`, the content script will keep the account in `blockedAuthors` and continue showing red tombstones until page reload. |
| Phase 19 action | Add an `else if (oldStatus === 'blocked' && newStatus !== 'blocked')` branch in the `flaggedAccounts` onChanged handler: remove the entry from `blockedAuthors`, and either remove existing red tombstones for that author's hidden posts or revert them to grey pending-tombstones. |
