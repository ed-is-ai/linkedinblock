  <what-built>A blank `DOM-INSPECTION.md` template ready to receive selector values. Claude cannot perform this step — it requires a logged-in LinkedIn session in a real Chrome browser, which has no CLI/API equivalent.</what-built>
  <how-to-verify>
    Open Chrome, log in to LinkedIn, navigate to `https://www.linkedin.com/feed/`. Open DevTools (F12) → Elements tab. For each of the following, find the element in the DOM tree and capture the most stable selector that is NOT a CSS class name:

    1. **Feed container** — the element whose immediate children are post cards. Look for `[data-finite-scroll-hotkey-context]` first. If absent, look for `main`, `[role="main"]`, or a data-* attribute on the scrolling region. Record the selector and the exact attribute name + value seen.

    2. **Post card** — the root element of a single post in the feed. Look for `data-urn`, `data-id`, `data-activity-urn`, or a `data-test-id`. Record the selector and an example value (e.g., `urn:li:activity:7012345678901234567`).

    3. **Post URN attribute** — the attribute (on the post card or a child) holding the URN. Record attribute name and example value.

    4. **Post author name** — the element containing the post author's display name. Look for `[data-anonymize="person-name"]`, `[aria-label*="View"]`, or a semantic `<a>` linking to `/in/`. Record the selector and how to extract the visible text.

    5. **Sponsored / promoted marker** — what distinguishes a Promoted post? Look for `aria-label*="Promoted"`, `data-test-id*="promoted"`, or hidden text. Scroll the feed until at least one promoted post appears. Record selector.

    6. **Company page marker** — what distinguishes a post authored by a company page (vs a person)? Look at the author block: companies typically link to `/company/<slug>`, people to `/in/<slug>`. Record the distinguishing selector or URL pattern.

    Record findings in `.planning/phases/01-foundation/DOM-INSPECTION.md` using the template in resume-signal below. Include the LinkedIn build version if visible (`<meta name="treeID">` or footer build hash) so future drift can be detected.
