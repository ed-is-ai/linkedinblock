## Summary

Two files reviewed: `src/dashboard/index.tsx` (the Preact dashboard component) and `src/dashboard/dataManagement.ts` (its pure-function utility layer). The utility layer is solid — RFC 4180 CSV escaping is correct, cleanse logic is correctly keyed on `lastSeenAt`/`hiddenAt` timestamps, and the JSON export shape is coherent.

The dashboard component has one critical bug (blob URL revoked before the browser completes the download), one data-integrity issue (cleanse irrecoverably destroys all dismissed-account history regardless of the selected date), several logic errors in the chart, and a silent-failure path in the cleanse UI. ## Critical Issues

### CR-01: Blob URL revoked before download completes

**File:** `src/dashboard/index.tsx:100-101`

**Issue:** `URL.revokeObjectURL(url)` is called synchronously immediately after `a.click()`. A programmatic `click()` on a download link is fire-and-forget — the browser schedules the download asynchronously. On slower machines or large exports (e.g., many stored posts), the blob may not have been fetched by the time the URL is revoked, resulting in a silently broken or empty download. This affects all three export handlers (`handleExportJson`, `handleExportCsv`, `handleExportPostsCsv`).

**Fix:**
```typescript
function triggerDownload(content: string, filename: string, mimeType: string): void {
  const blob = new Blob([content], { type: mimeType });
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = filename;
  document.body.appendChild(a);  // required in Firefox
  a.click();
  document.body.removeChild(a);
  // Defer revocation — give the browser time to initiate the download
  setTimeout(() => URL.revokeObjectURL(url), 10_000);
}
```

## Warnings

### WR-01: `hasData` check excludes valid 100%-hidden days, showing false "no data" message

**File:** `src/dashboard/index.tsx:22`

**Issue:** `const hasData = values.some(v => v > 0)` treats any day where all visible posts were hidden (score = 0% net) as a zero, and if every day in the window has this value, the chart displays "No feed data yet" even though data exists and was correctly processed. The computed value of `Math.round(((seen - hidden) / seen) * 100)` correctly returns 0 when `hidden === seen`, but this is indistinguishable from "no data for that day" in the `hasData` guard.

A better signal for "has data" is whether any stat entry exists for the window, not whether the net percentage is above zero.

**Fix:**
```typescript
// Replace:
const hasData = values.some(v => v > 0);

// With: check whether any day in the window had posts seen
const hasData = days.some(date => {
  const s = byDate.get(date);
  return s !== undefined && s.seen > 0;
});
```

### WR-02: Cleanse permanently destroys all dismissed accounts regardless of date

**File:** `src/dashboard/index.tsx:127`

**Issue:** `handleClean` unconditionally sets `dismissedAccounts: []`, wiping the entire dismissed list even when the user is only cleansing data older than a specific date. A user who dismissed a false positive last week but is cleaning data from 2 years ago will silently lose that dismissal, causing previously-dismissed accounts to resurface in the review queue as if newly flagged.

The `window.confirm` message mentions "all dismissed entries" but users may not read this carefully, and the consequence (accounts re-appearing in review) is non-obvious.

**Fix:** Filter `dismissed` to keep entries whose corresponding account is being retained. Since `dismissedAccounts` is a `string[]` with no timestamp, retain any ID that appears in `keptAccounts`:

```typescript
// After filterCleansed:
const { keptAccounts, keptPosts } = filterCleansed(accounts, posts, cleanseDate);
const keptDismissed = dismissed.filter(id => id in keptAccounts);
await chrome.storage.local.set({
  flaggedAccounts: keptAccounts,
  storedPosts: keptPosts,
  dismissedAccounts: keptDismissed,
});
setDismissed(keptDismissed);
```

Also update `filterCleansed` in `dataManagement.ts` to accept and return the filtered dismissed list, keeping the function pure and testable.

### WR-03: Cleanse button enabled but silently no-ops when `cleansePreview` is null

**File:** `src/dashboard/index.tsx:253-257`

**Issue:** The "Confirm cleanse" button is enabled (`disabled={!cleanseDate}`) whenever `cleanseDate` is non-empty. However, `handleClean` guards with `if (!cleanseDate || !cleansePreview) return` on line 120, so if `cleansePreview` is null (e.g., the `onInput` handler set the date but `deriveCleanseCount` returned before setting preview, or the component re-rendered in an intermediate state), clicking the button silently does nothing. There is no indication to the user that the action was a no-op.

**Fix:** Disable the button when `cleansePreview` is null, not only when `cleanseDate` is empty:
```tsx
<button
  style={cleanseDate && cleansePreview ? s.destructiveBtn : s.destructiveBtnDisabled}
  disabled={!cleanseDate || !cleansePreview}
  onClick={handleClean}
>Confirm cleanse</button>
```

### WR-04: 30-day chart x-axis may render duplicate label at index 21 / n-1

**File:** `src/dashboard/index.tsx:41`

**Issue:** The 30-day label array is `[0, 7, 14, 21, n-1]` where `n === 30`, making `n-1 === 29`. Indices 21 and 29 are distinct, so there is no immediate duplication. However, the hardcoded index `21` is not semantically `n * 0.7` — if the window size ever changes or the array is shorter than 22 elements (which cannot happen today but is fragile), `days[21]` could be undefined. Currently `days[21]?.slice(5) ?? ''` silently produces an empty label rather than throwing, but the empty label still renders a `<text>` element at that position, producing a phantom tick.

More concretely: the expression `days[i]?.slice(5) ?? ''` on line 41 will silently produce an empty string if `i` is out of range. An empty string renders as an invisible text element — cluttering the SVG DOM and potentially interfering with accessibility tooling.

**Fix:**
```typescript
const xLabels = timeWindow === 7
  ? days.map((d, i) => ({ i, label: d.slice(5) }))
  : [0, Math.round(n / 4), Math.round(n / 2), Math.round((3 * n) / 4), n - 1]
      .filter((i, pos, arr) => arr.indexOf(i) === pos && i < n)  // deduplicate and bounds-check
      .map(i => ({ i, label: days[i].slice(5) }));
```

### WR-05: Y-axis zero label rendered outside SVG viewport

**File:** `src/dashboard/index.tsx:50`

**Issue:** For the tick at `v === 0`, the label y-position is `yOf(0) - 4`. `yOf(0)` equals `pT + cH = 8 + 68 = 76`. So the label is at `y = 72`. The SVG height `H = 100` and `pB = 24`, so the chart area bottom is at `y = 76`. With `dominantBaseline` absent (unlike the other ticks which use `dominantBaseline="middle"`), the text baseline is at 72. This is fine numerically, but the inconsistency with other ticks (which use `dominantBaseline="middle"`) means the 0% label aligns differently from the 50% and 100% labels. On the `maxY` tick (top), `yOf(maxY) = pT = 8`, and using `dominantBaseline="middle"` centers it at 8 — which may clip the top half of the text above the SVG viewport.

**Fix:** Use consistent baseline alignment and add a small top-padding guard:
```tsx
<text
  x={pL - 8}
  y={yOf(v)}
  fontSize={9}
  fill="#9ca3af"
  textAnchor="end"
  dominantBaseline={v === maxY ? 'hanging' : v === 0 ? 'auto' : 'middle'}
>
  {v}%
</text>
```

## Info

### IN-01: `signals` column in CSV export embeds JSON — poor machine readability

**File:** `src/dashboard/dataManagement.ts:55`

**Issue:** `JSON.stringify(a.signals)` produces a JSON object string (e.g., `{"aiLanguage":45,"botBehavior":30}`) which `csvEscape` wraps in double-quotes. The resulting CSV field is parseable per RFC 4180, but downstream tools (Excel, pandas) will treat it as an opaque string. Users who want per-signal data cannot easily work with it.

**Fix:** Either flatten signals into dedicated columns (one column per known signal key), or document this limitation in a `README` or in the export filename/header so users know they need a secondary parse step.

### IN-02: Non-null assertion on `document.getElementById('root')` with no fallback

**File:** `src/dashboard/index.tsx:315`

**Issue:** `document.getElementById('root')!` will throw a `TypeError` at runtime if the element is absent (malformed `dashboard.html`). The error will surface as an unhandled exception with no user-visible message.

**Fix:** Add a guard:
```typescript
const root = document.getElementById('root');
if (!root) throw new Error('Dashboard root element not found. Check dashboard.html.');
render(<App />, root);
```
This produces the same throw but with a descriptive message that aids debugging.

### IN-03: `loadError` state set but never cleared on retry

**File:** `src/dashboard/index.tsx:78, 89-91`

**Issue:** `loadError` is displayed in the Data management card. There is no way for the user to retry the load or dismiss the error — the UI message says "Try reopening the dashboard" but there is no retry button. Because the `useEffect` runs once on mount with an empty dependency array, the only recovery path is to close and reopen the page. If a transient `chrome.storage.local` error occurs (which is rare but possible under storage quota pressure), the user has no in-page recovery.

**Fix:** Add a retry mechanism:
```tsx
const [loadAttempt, setLoadAttempt] = useState(0);
// In useEffect, include loadAttempt in the dependency array
useEffect(() => { /* same load logic */ }, [loadAttempt]);
// In the error display:
{loadError && (
  <div style={s.errorMsg}>
    {loadError} <button onClick={() => setLoadAttempt(n => n + 1)}>Retry</button>
  </div>
)}
```
