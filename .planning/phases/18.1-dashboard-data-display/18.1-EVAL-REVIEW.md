## Context: What This Phase Is

Phase 18.1 delivered UI changes to a Chrome extension dashboard (Preact component). There is no generative AI in the dashboard itself — it is a deterministic display layer that reads from `chrome.storage.local` and renders statistics. The AI component of this project is the heuristic/ML detection engine in the content script. This distinction is critical: standard AI eval dimensions (hallucination, faithfulness, LLM judges) do not apply to the dashboard UI layer directly. The applicable eval dimensions are those governing **output structure validity**, **task completion**, and **safety** in the context of a deterministic UI component, plus the product-level concern of whether the dashboard accurately represents the AI detection system's outputs to the user.

## Critical Gaps (BLOCKER)

### BLOCKER-01: No CI/CD pipeline — eval cannot gate deployments

**Planned:** Not explicitly planned in AI-SPEC (absent), but the presence of Vitest and `npm run type-check` implies these should run automatically.
**Found:** Zero CI/CD configuration. No `.github/workflows/`. No automated run of `type-check`, `build`, or Vitest on any code push.
**Impact:** A regression in any subsequent phase can break the build without detection until a human manually runs the check.
**Remediation:** Add `.github/workflows/ci.yml` with: `npm run type-check && npx vitest run && npm run build`. This closes the gap with a single file.

---

### BLOCKER-02: `NetPostsChart` has zero unit test coverage — confirmed bug undetected

**Planned:** Phase plan task acceptance criteria include chart math correctness implicitly (the chart must render valid percentages).
**Found:** `src/dashboard/index.tsx` has no test file. The `hasData` bug (WR-01 in REVIEW.md) — where a chart full of 0% values falsely shows "No feed data yet" — is confirmed in code review but is not caught by any test. The bug is live in the build.
**Impact:** Users who have actively used the extension but happen to have every visible post hidden (100% filter rate) see a false "no data" message. This directly undermines the dashboard's stated purpose: showing feed health.
**Remediation:** Add `src/dashboard/NetPostsChart.test.ts` with jsdom environment. Minimum cases: (1) all values = 0 with `seen > 0` data in byDate — assert chart renders SVG, not the no-data message; (2) values array with mixed data; (3) empty `stats` array. Also fix the `hasData` guard per WR-01 before adding the test.

---

### BLOCKER-03: Data-loss path in `handleClean` has no test and a confirmed bug

**Planned:** Cleanse UI was pre-existing; phase 18.1 did not modify it. However, the build includes it and the REVIEW audit found it.
**Found:** `handleClean` (index.tsx:127) sets `dismissedAccounts: []` unconditionally, destroying all dismissed-account history on any cleanse operation. No test covers `handleClean` logic. `filterCleansed` in dataManagement is tested, but the integration of that function into the actual storage write is not.
**Impact:** A user who runs a date-filtered cleanse loses all their manual false-positive dismissals. Those accounts re-enter the review queue as newly flagged. This is an irreversible data-loss event.
**Remediation:** Fix WR-02 per REVIEW.md (filter dismissed IDs against keptAccounts). Add a test for the cleanse integration behaviour: mock `chrome.storage.local`, call `handleClean`, assert `dismissedAccounts` in storage contains only IDs still present in keptAccounts.

## Remediation Plan

### Must fix before production:

1. **Add CI/CD** (BLOCKER-01): Create `.github/workflows/ci.yml`. Steps: checkout, `npm ci`, `npm run type-check`, `npx vitest run`, `npm run build`. Effort: 30 minutes.

2. **Fix and test `hasData` guard** (BLOCKER-02, WR-01): Change guard from `values.some(v => v > 0)` to checking whether any `byDate` entry has `seen > 0`. Add `NetPostsChart.test.ts` with the 3 minimum cases above. Effort: 2 hours.

3. **Fix and test cleanse dismissed-accounts wipe** (BLOCKER-03, WR-02): Apply the `keptDismissed` filter in `handleClean`. Add integration test. Effort: 2 hours.

4. **Fix blob URL revocation** (CR-01 from REVIEW): Wrap all three download handlers in a `triggerDownload` utility with `setTimeout(() => URL.revokeObjectURL(url), 10_000)`. Add a test that mocks `URL.createObjectURL` and asserts revocation does not happen synchronously. Effort: 1 hour.

### Should fix soon:

5. **Add `test` script to package.json**: `"test": "vitest run"`. Without this, `npm test` fails — a standard CI expectation. Effort: 5 minutes.

6. **Disable cleanse button when `cleansePreview` is null** (WR-03): Change button's disabled condition. Effort: 10 minutes.

7. **Add human-independent toggle test**: Use jsdom + Preact test utilities to mount `App` with mock storage and assert the 7/30-day toggle updates rendered label content. This removes the sole reliance on the unverifiable human checkpoint. Effort: 3 hours.

### Nice to have:

8. **WR-04** (phantom SVG tick for out-of-range label): Apply bounds-check filter to 30-day label array. Effort: 20 minutes.

9. **WR-05** (y-axis baseline inconsistency): Apply `dominantBaseline` conditional per REVIEW fix. Effort: 10 minutes.

10. **IN-01** (signals column is opaque JSON in CSV): Flatten signals into dedicated columns or document the limitation. Effort: 1 hour.
