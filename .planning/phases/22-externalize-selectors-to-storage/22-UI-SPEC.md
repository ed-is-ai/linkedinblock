# Phase 22 — UI Design Contract

> Visual and interaction contract for the selector health view and reset control added to the
> dashboard in Phase 22: Externalize Selectors to Storage.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Scope

Two UI surfaces are introduced in this phase:

1. **SelectorView panel** — a read-only diagnostic panel permanently visible in the dashboard,
   listing each selector target's active selector string, source badge, last-matched info, and a
   session-miss warning for feed-essential targets.

2. **Reset-to-defaults control** — a two-step inline confirmation button inside the SelectorView
   panel. Reuses the BatchBlockBar confirm-step pattern from Phase 20.

No changes are made to the popup (compact path), feed behavior, detection scoring, or any
existing dashboard panel.

**Accent reserved for:** Reset-now confirmation button fill; `seed` source badge color; count

names.

**Warning color (`#dc2626`) reserved for:** Feed-essential target rows with a session miss only.
This maps to the four targets declared in CONTEXT.md D-03: `FEED_CONTAINER`, `POST_CARD`,

switch to `#dc2626` when the session-miss condition is true.

**Neutral stale (`#9ca3af`) used for:** Contextual/optional target rows where the selector was
not seen in the current session. No border change. This is informational, not alarming.

## Component Inventory

### SelectorView (new — `src/dashboard/SelectorView.tsx`)

A card panel matching the existing `s.card` pattern in dashboard/index.tsx:
- `background: '#fff'`, `border: '1px solid #e5e7eb'`, `borderRadius: 8`,
  `padding: '20px 24px'`, `marginBottom: 16`
- Panel heading ("Selector Health") uses `s.cardHeading` style:
  `fontSize: 13, fontWeight: 600, color: '#374151', marginBottom: 12`

**Table layout** (no HTML `<table>`; use `div` rows with flex):

| Column | Width | Content |
|--------|-------|---------|
| Target | ~30% | Target name (`FEED_CONTAINER`, `POST_CARD`, etc.) at 13px/400 |
| Active selector | ~40% | Current selector string at 12px/400, truncated with `textOverflow: 'ellipsis'` |
| Source | ~15% | Source badge (see below) |
| Last matched | ~15% | ISO date string at 12px/400, muted; "—" if never matched |

Each row: `display: 'flex'`, `alignItems: 'center'`, `gap: 8`, `padding: '6px 0'`,
`borderBottom: '1px solid #e5e7eb'`.

**Session-miss warning (feed-essential targets):** When a feed-essential target produced zero
matches this session, apply to the entire row:
- All text switches to `color: '#dc2626'`
- Add `borderLeft: '2px solid #dc2626'` and `paddingLeft: 6` to the row div
- Do NOT add any icon or emoji — text color change + border is the full signal

**Session-miss neutral (contextual targets):** When a contextual target produced zero matches:
- Target name and last-matched columns switch to `color: '#9ca3af'`
- Append the annotation "(not seen this session)" at 11px/400 in `#9ca3af` after the last-matched value
- No border change

**Column header row:** `fontSize: 12, color: '#6b7280', fontWeight: 600, paddingBottom: 8,
borderBottom: '1px solid #e5e7eb', marginBottom: 4` — matches `s.metricLabel` pattern.

### Source Badge

A small inline chip for the `source` column:

| Source value | Label | Color |
|---|---|---|
| `seed` | `seed` | `color: '#0a66c2'`, `background: '#eff6ff'` |
| `heuristic` | `heuristic` | `color: '#374151'`, `background: '#f3f4f6'` |
| `llm` | `llm` | `color: '#6b7280'`, `background: '#f9fafb'` |
| `user` | `user` | `color: '#374151'`, `background: '#f3f4f6'` |

Badge style: `fontSize: 10, padding: '2px 6px', borderRadius: 10` — matches existing `chip`
pattern from AccountRow.tsx.

In Phase 22, only `seed` will appear in practice (no healed candidates yet). The heuristic/llm
badges are specified now so Phase 23 can render them without touching this contract.

### Reset-to-Defaults Control (inline confirm, new)

Located at the bottom of the SelectorView card, below the target table, separated by a
`<hr style={{ margin: '16px 0', border: 'none', borderTop: '1px solid #e5e7eb' }} />`.

**Idle state** — single button matching `s.actionBtn` from dashboard:
```
padding: '6px 16px', border: '1px solid #d1d5db', borderRadius: 6,
background: '#fff', cursor: 'pointer', fontSize: 13
```
Label: `Reset to defaults`

**Confirming state** — morphs into a confirm strip matching `barStyles.confirmStrip` from
BatchBlockBar.tsx. No browser `confirm()` dialog. State machine: `idle → confirming → idle`
(on cancel) or `idle → confirming → writing → idle` (on confirm).

Confirm strip layout:
```
background: '#f3f4f6', borderRadius: 4, padding: '8px'
```

Message: `Reset all selectors to bundled defaults?` at `fontSize: 12, color: '#374151'`.

Button row (`display: 'flex', justifyContent: 'space-between', gap: 8, marginTop: 8`):

- Cancel button: `padding: '6px 12px', background: '#f3f4f6', color: '#374151',
  border: '1px solid #d1d5db', borderRadius: 4, cursor: 'pointer', fontSize: 12`
  Label: `Cancel`

- Confirm button (idle): `flex: 1, padding: '6px 0', background: '#0a66c2', color: '#fff',
  border: 'none', borderRadius: 4, cursor: 'pointer', fontSize: 13, fontWeight: 600`
  Label: `Reset now`

- Confirm button (writing/disabled): same + `opacity: 0.7, cursor: 'not-allowed'`
  Label: `Resetting…`

**After successful reset:** the component exits the confirming state and returns to idle. The
health table reflects the restored seed values immediately because `chrome.storage.onChanged`
triggers a re-read of the registry (SELECTOR-08). No separate success toast or banner needed.

## States and Interactions

### SelectorView Panel States

| State | Trigger | Rendering |
|-------|---------|-----------|
| Loading | Component mount before storage read completes | "Loading selector health…" at `fontSize: 12, color: '#9ca3af'` centered in the card |
| Populated — all healthy | Registry loaded, no session misses on feed-essential targets | Normal table, all rows default text color |
| Populated — stale warning | One or more feed-essential targets have zero session matches | Affected rows red + left border; warning count shown in panel heading: `"Selector Health (N stale)"` at `color: '#dc2626'` |
| Populated — contextual miss | Only contextual targets are session-misses | Normal panel heading; affected rows show grey annotation |
| Error | `chrome.storage.local.get` rejects | "Could not load selector health. Try reopening the dashboard." at `fontSize: 12, color: '#dc2626'` — matches `s.errorMsg` |

### Reset Control States

| State | Transition |
|-------|-----------|
| Idle | Default; shows "Reset to defaults" button |
| Confirming | User clicked "Reset to defaults" → morphs to confirm strip |
| Writing | User clicked "Reset now" → button label becomes "Resetting…", disabled |
| Idle (post-reset) | Write completes → returns to idle; table reflects new state via onChanged |
| Error (post-reset) | Write fails → return to idle; show error message below the strip: "Reset failed. Try again." at `fontSize: 12, color: '#dc2626'` |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Panel heading (healthy) | `Selector Health` |
| Panel heading (stale warning) | `Selector Health (N stale)` — N = count of feed-essential stale targets |
| Column headers | `Target` / `Active selector` / `Source` / `Last matched` |
| Loading state | `Loading selector health…` |
| Error state | `Could not load selector health. Try reopening the dashboard.` |
| Stale row annotation | Row text color change only; no separate annotation text for feed-essential |
| Contextual miss annotation | `(not seen this session)` appended after last-matched value |
| Reset idle CTA | `Reset to defaults` |
| Reset confirm message | `Reset all selectors to bundled defaults?` |
| Reset cancel label | `Cancel` |
| Reset confirm label (idle) | `Reset now` |
| Reset confirm label (writing) | `Resetting…` |
| Reset error message | `Reset failed. Try again.` |
| Never-matched last-matched | `—` (em dash) |
| Session-match present | ISO date string: `YYYY-MM-DD` |

**Destructive action:** "Reset to defaults" is a reversible destructive action (storage write
that discards any healed candidates). It uses the inline confirm step — NOT `window.confirm()`.
No modal. No navigation. Confirmation copy states the consequence clearly without jargon.

## Placement in Dashboard

The SelectorView card is inserted **after** the existing feed-health stats cards and **before**
the "Data management" card. Insertion point in `dashboard/index.tsx`:

```
[heading + toggle]
[Net AI voice posts card]       ← existing
[Posts hidden / bar metrics card] ← existing
[Selector Health card]          ← NEW (this phase)
[Data management card]          ← existing (unchanged)
```

This placement keeps diagnostic/system panels grouped before the destructive data-management
section, and keeps the SelectorView near the feed-health context it supports.

## Implementation Notes for Executor

1. **File to create:** `src/dashboard/SelectorView.tsx` — self-contained component, props-fed
   from dashboard App state. No direct storage access inside SelectorView; parent reads from
   `chrome.storage.local` and passes `entries` + `sessionMisses` as props.

2. **Wire in `dashboard/index.tsx`:** Add a `selectorRegistry` state slice, load it alongside
   existing storage reads in `useEffect`, and subscribe to `chrome.storage.onChanged` for
   `selectorRegistry` key (same pattern as the existing `flaggedAccounts` listener).

3. **Reset handler lives in dashboard App:** `handleResetSelectors()` writes the seed defaults
   back to `selectorRegistry` key in `chrome.storage.local`. SelectorView receives this as an
   `onReset: () => Promise<void>` prop, matching the `onBatchBlock` prop shape of BatchBlockBar.

4. **Session-miss data:** SelectorView receives `sessionMisses: Set<SelectorTarget>` injected
   by the content script via storage (key: `selectorSessionMisses`, written by SelectorRegistry
   on each page session). Dashboard reads this key alongside `selectorRegistry`.

5. **Feed-essential targets (hard-coded in this component):**
   `['FEED_CONTAINER', 'POST_CARD', 'POST_AUTHOR_LINK', 'POST_BODY_TEXT']`
   Any target in this list that appears in `sessionMisses` triggers the red/stale warning.
   All other targets in `sessionMisses` trigger the neutral grey annotation only.

6. **No new dependencies.** No icon library, no utility library, no CSS modules.
